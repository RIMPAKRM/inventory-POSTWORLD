# Архитектура: кастомный инвентарь, экипировка и крафт мод forge 1 20 1

## 1. Назначение документа
Документ фиксирует целевую архитектуру для разработки через Copilot Pro.

Фокус только на подсистеме:
- кастомный инвентарь в стиле DayZ (емкость зависит от экипировки);
- замена ванильного инвентаря (кроме creative);
- кастомный экран крафта с категориями и карточками рецептов;
- только рецепты этого мода.

## 2. Обновленные продуктовые решения
1. `CHEST` и `VEST` - независимые слои экипировки.
2. `FACE` и `GLOVES` не дают storage-слоты.
3. Overflow при снятии экипировки: сначала перенос в доступные активные слоты, затем дроп на землю.
4. Сервер - единственный источник правды; клиент только запрашивает действия и отображает результат.
5. Инвентарь реализуется через стандартные Forge/vanilla `Slot` и `AbstractContainerMenu`.
6. Совместимость с другими модами - обязательный приоритет.
7. Механика `death -> corpse` переносится на финальный этап разработки.
8. Защита (`armor`) должна быть разной для разных предметов и масштабируемой под будущий контент.

## 3. Scope
### In scope
- Слоты экипировки: `HEAD`, `FACE`, `CHEST`, `VEST`, `BACKPACK`, `GLOVES`, `LEGS`, `FEET`.
- Динамические storage-слоты от экипировки (минимум: `VEST`, `BACKPACK`).
- Кастомный `InventoryScreen` вокруг превью персонажа.
- Кастомный `CraftScreen` (вкладки + карточки рецептов).
- Серверная валидация всех инвентарных и крафтовых операций.
- Data-driven модель для экипировки и крафтов.
- Масштабируемая система параметров защиты предметов.

### Out of scope
- Радиация, оружие, строительство баз, репутация, бестиарий и другие подсистемы.
- Полная кросс-мод интеграция на уровне API других модов (кроме совместимости стандартного инвентарного поведения).

## 4. Ключевые требования
- В survival/adventure у игрока доступен кастомный инвентарь; в creative - ванильный экран.
- Без экипировки у игрока остается только хотбар.
- Активные storage-слоты определяются текущей экипировкой.
- Все изменения состояния инвентаря и крафта выполняются только на сервере.
- Логика слотов и взаимодействия должна быть максимально ванильной по поведению.
- Крафтовый UI использует только рецепты этого мода.

## 5. Доменная модель
### 5.1 Сущности
- `PlayerLoadout`: состояние экипировки и динамических storage-слотов.
- `EquipmentSlotType`: типы экипировочных слотов.
- `StorageProfile`: описание емкости, которую дает конкретный предмет экипировки.
- `ProtectionProfile`: параметры защиты предмета.
- `CraftCategory`: категория крафтов.
- `CraftCard`: карточка крафта (ингредиенты, результат, условия).

### 5.2 Модель защиты (масштабируемая)
`ProtectionProfile` задается на предмет и может расширяться без переписывания ядра.

Базовые поля:
- `armorValue`;
- `durabilityModifier`;
- `weightClass`;
- `tags`.

Контракт для `tags`:
- теги имеют namespace (`modid:tag_name`);
- конфликтующие теги разрешаются через `priority` профиля;
- порядок применения детерминирован: сортировка по `priority`, затем по `resourceLocation`.

### 5.3 Инварианты
- Нельзя класть предмет в неактивный слот.
- Нельзя класть предмет в слот неподходящего типа.
- При деактивации слотов применяется фиксированный алгоритм overflow.
- Любая мутация инвентаря валидируется сервером.

## 6. Транзакционная модель и защита от гонок
### 6.0 Комбинация паттернов
Для MP используется комбинированный подход:
- `Saga` (оркестрация/хореография) для длинных многошаговых операций (`overflow`, `craft`, позднее `corpse/loot`);
- `per-player lock + loadoutVersion` для локальной сериализации действий игрока;
- `idempotency tokens` для C2S запросов;
- атомарные NBT-транзакции для сохранения/восстановления;
- `debounce + batch` для S2C синхронизации.

### 6.1 Атомарность операций
Каждая серверная операция инвентаря выполняется как транзакция:
1. `begin` (проверка `loadoutVersion`, блокировка игрока);
2. `plan` (расчет перемещений/дропа);
3. `commit` (применение к инвентарю и миру);
4. `sync` (отправка итогового состояния);
5. `unlock`.

### 6.2 Конкурентность
- Все операции инвентаря идут через per-player lock/queue.
- Параллельные клики/крафт не исполняются одновременно.
- Любая операция проверяет актуальность `loadoutVersion`; при mismatch отклоняется и запрашивает повтор.

### 6.3 Idempotency C2S
- Каждый `C2S_ClickCustomSlot` и `C2S_RequestCraft` содержит `requestId`.
- На сервере работает dedup-cache с TTL.
- Повторный `requestId` не приводит к повторному списанию/выдаче предметов.

### 6.4 Partial stack в overflow
- Перенос выполняется по стекам детерминированно: сначала полное слияние, затем частичное.
- Остаток стека, который не поместился, переходит в шаг `dropToWorld`.
- Операция не допускает двойной судьбы одного и того же предмета (или перемещен, или дропнут).

### 6.5 Recovery при relog/disconnect
- Состояние активной overflow-транзакции пишется в capability/NBT как `pendingOverflow`.
- При входе игрока сервер завершает или откатывает незавершенную транзакцию по журналу шагов.

### 6.6 Saga и компенсации
- Для операций с несколькими стадиями используется Saga-контур.
- Оркестратор хранит шаг, статус и список компенсаций.
- Ошибка на шаге запускает компенсацию в обратном порядке (`releaseReservation`, `rollbackMove`, `cancelDropMark`).
- Все переходы Saga логируются с `opId`.

## 7. Контракт слотов и совместимость
### 7.1 Canonical mapping
- Ванильные слоты (hotbar/armor/offhand) имеют фиксированные индексы.
- Динамические слоты имеют отдельный диапазон индексов и не смещают ванильные.
- Нельзя менять индексный контракт между патчами без миграции.

### 7.2 Признак активности
- Для каждого динамического слота хранится `slotActive:boolean`.
- Неактивный слот для внешних систем трактуется как недоступный, а не как новый тип контейнера.

### 7.3 Поведение для сторонних модов
- Чтение vanilla-совместимых диапазонов всегда стабильно.
- Дополнительные слоты доступны как расширение, но не ломают базовое ожидание модов от инвентаря игрока.

## 8. Архитектура по слоям
### 8.1 Common/Core
- Правила слотов, емкости, ограничений.
- Overflow-сервис: `moveToAvailable -> dropToWorld`.
- Правила применения `ProtectionProfile`.
- Алгоритмы распределения стеков и детерминированный порядок обработки.

### 8.2 Data
Data-driven ресурсы:
- `storage_profiles/*.json`;
- `protection_profiles/*.json`;
- `craft_categories/*.json`;
- `craft_cards/*.json`.

Правила валидации и fallback:
- каждый файл валидируется по схеме;
- некорректный объект отключается точечно (soft-disable), мир продолжает загрузку;
- в лог пишется error code, путь файла и причина;
- при необходимости применяется безопасный default-профиль.

### 8.3 Server
- Capability игрока (`PlayerLoadout`) + NBT сериализация.
- `schemaVersion` обязателен с первой версии capability.
- Миграции данных выполняются при загрузке игрока.
- Атомарная запись capability: `prepare -> write temp -> commit marker`.
- При аварийном завершении используется recovery по маркеру последней успешной транзакции.
- `AbstractContainerMenu` с проверкой кликов/переносов.
- Крафт-обработчик: `validate -> reserve -> consume -> grant`.
- Пересчет активных слотов при изменении экипировки.

### 8.4 Network
Минимальный набор пакетов:
- `C2S_ClickCustomSlot` (`requestId`, `clientViewVersion`);
- `C2S_RequestCraft` (`requestId`);
- `S2C_LoadoutSync` (`serverVersion`);
- `S2C_CraftDataSync` (если требуется).

Стратегия синхронизации:
- агрегация изменений и debounce отправки (`50-200ms`, конфиг);
- мгновенная отправка только для критических событий (commit overflow, завершение craft).

### 8.5 Client/UI
- Экран инвентаря: превью игрока в центре, слоты вокруг, хотбар всегда видим.
- Locked/disabled состояние недоступных слотов.
- Экран крафта: категории, карточки, отображение доступности рецепта.
- Creative mode: fallback на ванильный UI.
- Клиентские оптимистичные анимации помечаются как pending и откатываются при серверном отказе.

## 9. Критические потоки
### 9.1 Экипировка и динамические слоты
1. Игрок меняет экипировку.
2. Сервер берет lock игрока и увеличивает `loadoutVersion`.
3. Сервер пересчитывает емкость по `StorageProfile`.
4. Если емкость снизилась, выполняет overflow-транзакцию.
5. Сервер коммитит итог и отправляет `S2C_LoadoutSync`.

### 9.2 Крафт
1. Клиент отправляет `RequestCraft(requestId)`.
2. Сервер проверяет dedup и валидирует рецепт.
3. Сервер атомарно списывает ресурсы и выдает результат.
4. Сервер синхронизирует итоговое состояние.

## 10. Надежность, логирование и метрики
- Сервер - единственный источник правды.
- Любые перемещения проверяются на доступность и тип слота.
- Крафт и overflow выполняются атомарно.
- Логируются отклоненные запросы, повторы `requestId`, rollback/recovery.
- Метрики: количество overflow, доля rejected-операций, частота rollback, средний размер sync-пакетов.
- Для расследования дюпов хранится короткий trace по `opId` (несколько последних шагов).

## 11. Серверные API-контракты
- `beginLoadoutOp(player): OpContext` / `endLoadoutOp(player, success)` - захват/освобождение per-player lock и работа с `loadoutVersion`.
- `validateAndApplyClick(player, slotId, opId)` - серверная валидация и применение клика с idempotency.
- `applyOverflow(player, items, targetSlots, opId)` - транзакционный `moveToAvailable -> markDrop -> dropToWorld`.
- `requestCraft(player, craftId, requestId)` - проверка, резервирование стеков, атомарный `consume/grant`, компенсация при ошибке.
- `serializeLoadoutNBT(player)` / `migrateLoadoutNBT(nbt, schemaVersion)` - сохранение и миграция capability.

## 12. Тестовая стратегия
### Unit/logic
- `PlayerLoadout` save/load + миграции `schemaVersion`.
- Overflow: полное место, частичное место, нулевое место.
- Partial stack распределение.
- Idempotency `requestId`.
- Валидация `ProtectionProfile` и разрешение конфликтующих `tags`.

### Concurrency
- Параллельные сценарии `equip + shift-click + craft`.
- Повторные C2S запросы с разным порядком доставки.
- Disconnect/relog в середине overflow.

### Property-based
- Инварианты после любой последовательности операций: нет отрицательных стаков, нет дубликатов одного и того же `opId`-результата, сумма предметов сохраняется с учетом дропа.

### Fuzz/data
- Fuzz JSON для `craft_cards` и `protection_profiles`.
- Проверка graceful handling и soft-disable некорректных объектов.

### Smoke SP/MP
- `equip -> relog -> state restore`.
- `rapid move/shift-click`.
- `unequip with full dynamic slots`.
- `creative vs survival inventory UI`.
- Совместимость с модом, который читает инвентарь как vanilla.
